优先预加载机制

只接收图片clip时，主页滑动流畅，加载快；自从开始接收视频clip后，主页滑动卡顿严重加剧，推文封面很慢。添加预加载机制，预先加载可见项前后的视频封面

  • 在RecyclerView滚动回调里触发preloadVideoCovers并用lastPreloadRange做防抖，避免重复启动过多任务

  • 用preloadedUrls去重，避免重复请求同一URL

  • 使用独立的CoroutineScope并指定Dispatchers.IO，使预加载尽快在线程池的IO线程上执行

  • 用 Semaphore（videoPreloadSemaphore）限制并发，防止和UI请求互相争抢太多带宽/线程，从而保证每个预加载能完整执行

  • 对Glide请求调用submit后在后台线程上get阻塞等待结果，强制把资源下载/解码并写入Glide的磁盘/内存缓存，后续UI绑定会直接命中缓存

  • Glide仍会根据内部队列调度请求，所以显式设置请求优先级Priority.IMMEDIATE

尽可能减少视频封面解码

多轮优化后，引入视频clip带来的卡顿仍然明显。考虑到在现实场景中，视频封面应当会事先保存在服务器中，而不是让客户端自行解码；再者，用户不会点进每一篇推文查看，更重要的是滑动的流畅度。这里最终选择了尽量用图片clip直接作为封面图，而不去解析第一个视频clip作为封面的方案。从头到尾检索推文中所有clip，用第一个图片clip作为封面图，只有在clips中没有图片只有视频的时候才去解码视频封面。这个优化大大提升了滑动体验

3. @Inject

@Inject是最常用的Mixin注解，用于向指定方法中插入自定义代码。该注解的两个常用参数为method和at：

  • method：注入的目标方法名。
  • at：注入的位置，格式为@At(...)，有以下几种注入位置可选：
标识 描述
INVOKE 方法调用时
HEAD 方法头部
RETURN 方法返回前（每个return前）
TAIL 方法尾部（最末return前）

  • public：访问修饰符无严格要求，一般与原方法相同。
  • void：返回类型硬性要求填void，因为这个注入方法不会被获取返回值。
  • handleItem：注入方法的方法名可以任选，无影响。
  • CallbackInfo：方法返回参数，必填，可用于终端原方法或修改返回逻辑。

物品入水：乜又係你啊陈生

我们又回到了上次的那个问题：玩家扔出一个物品，要检测物品是否掉进水中。如果用TickEvent来持续检测，所增加的持续性开销太大了。

简单的想法是，玩家抛出物品时开启一个计时器，比如两秒钟之后，才执行仅此一次的检测逻辑。

如何实现？

沿用Tick作为计时工具

引入一个新情景：玩家右键使用物品时，将玩家的游戏模式调整为旁观者模式；计时60秒后，恢复为原来的游戏模式。

最不具侵入性的方法是，实现一个数据结构，记录Runnable和延迟时间，订阅TickEvent，并不断减少数据结构中每个任务的延迟时间（距离执行任务所剩时间），当剩余时间小于等于0时，执行对应的Runnable

注意点

  • 只有在逻辑服务端线程上执行的逻辑才有意义。最优获取方法：ServerLifecycleHooks.getCurrentServer()，返回类型为MinecraftServer 执行方法是#execute。
    调用Level#getServer得到的实例的execute方法有时是无效的，因为根据执行环境的不同，这种方法得到的未必是逻辑服务端线程。
  • Map相关的操作中，即使Minecraft的主要逻辑处理都在单线程上执行，也需要用到ConcurrentHashMap，以预备未来的扩展需求或潜在的并发修改；删除Map中元素时使用迭代器：iterator.remove()来避免ConcurrentModificationException

遗憾的是，这种方法确实增加了持续性开销。如果我们死都不想用TickEvent怎么办？

梳理一下实现逻辑：

  1. TFTickHandler中，subscribe了PlayerTickEvent.Post

  2. 对于每个玩家，默认每20tick触发checkForPortalCreation方法（此前还要判断是否是ServerPlayer、是否有portalCreationPermission等等，此处略）

我的第一直觉：这个写法的性能影响真的呆胶布？？？

首先，对于每个玩家都会subscribePlayerTickEvent，也就是在线玩家越多，性能开支越大

其次，每秒遍历玩家一定半径中的所有掉落物，显然是不合理的。

暮色传送门的开启，一个玩家或许只会进行一次。你是说，为了实现这仅此一次的逻辑，玩家需要背负这么大的持续性能开销？

针对性地修改判定触发逻辑：我补药用TickEvent啊

首先，谁会想用投掷器来开传送门？？为了只检测玩家扔出的物品，我直接先subscribe了ItemTossEvent

直接的想法是，预定一个延迟判断事件，比如在物品被扔出后的五秒后，进行判断，仅此一次

Minecraft NeoForge 事件系统

发表于 2025-05-07

使用NeoForge 21.5.42-beta，支持Minecraft 1.21.5
参考：NeoForge文档::Events

1. 事件系统与事件总线

事件系统是NeoForge/Forge的主要功能，用于捕获事件并执行自定义逻辑。事件是指游戏中发生的事件，如玩家跳跃、挖掘、登入登出、资源加载等等。
开发者可以注册(Subscribe)一个事件(Event)，甚至可以自定义新的事件。

事件总线(Event Bus)是事件被处理的地方，在NeoForge中有两个事件总线：

  • GAME总线(Forge中称为Forge总线)：生命周期为世界加载到世界关闭，是处理游戏中逻辑的最常用总线，即NeoForge.EVENT_BUS
  • MOD总线：生命周期为mod加载到游戏关闭， 常用于资源处理。该总线中的事件会并行执行

2. 注册事件处理方法

事件处理方法(Event Handler)都采用方法来实现，要求该方法：

  1. 只传入一个事件作为参数
  2. 返回值为void

事件的注册有几种方法：

  • IEventBus#addListener：在Mod主类构造方法里调用对应总线的addListener方法，传入类名::方法名

  • @SubscribeEvent注解：在处理方法上加这一注解，然后在Mod主类构造方法调用addListener并传入对应类的实例(即addListener(new 类名()))，将自动注册所有注解了@SubscribeEvent的方法。如果方法都为静态方法，则可以传入类本身(即addListener(类名.class))，自动注册所有静态方法

  • @EventBusSubscriber注解：最佳实践。只需为类注解@EventBusSubscriber，NeoForge将自动找到该类，并将该类下所有注解了@SubscribeEvent的静态方法注册到事件总线中。因此，事件处理方法的要求可加上一条：需要是静态方法。该注解可以添加三个参数:modid、bus、value，分别传递模组ID、事件总线类型、注册环境

    • modid：传入MOD_ID，便于debug时检查哪个mod的事件注册发生错误
    • bus: 传入EventBusSubscriber.Bus.GAME或EventBusSubscriber.Bus.MOD以指定注册的总线
    • value：传入Dist.CLIENT或Dist.DEDICATED_SERVER，指定注册的物理环境。必须注意，Level#isClientSide()检查的是逻辑环境。
      比如，客户端启动单人游戏，是物理客户端上存在逻辑客户端和逻辑服务端；而服务端启动服务器，才存在物理服务端。

3. 事件结构

3.1 字段与方法

事件的成员包括字段和方法，包含了事件的发起者、目标、环境，还有可以直接执行的操作等等，应根据不同事件的不同成员区别使用。

3.2 继承结构

所有的事件都有公共父类Event，其下方又有许多子类。如以下继承链：Event - EntityEvent - LivingEvent - PlayerEvent - PlayerTickEvent

3.3 优先级

NeoForge/Forge用EventPriority枚举类为事件处理方法定义了五级优先级：HIGHEST,HIGH,NORMAL(默认值),LOW,LOWEST，可以在@SubscribeEvent注解或IEventBus#addListener中定义。
事件处理方法将按照优先级从高到低被执行。注意，对于并行处理的事件，优先级无效。

3.4 取消

所有实现了ICancellableEvent接口的事件都可以被取消，拥有#setCancel(boolean canceled)和#isCanceled()方法。事件被取消后，其它事件处理方法不会被执行。
比如，取消玩家的LivingDeathEvent能防止玩家死亡(还需回复玩家血量，否则会由于血量为0卡在死亡界面)；取消生物的LivingChangeTargetEvent可以让该生物的目标实体保持不变。

事件处理方法还可以只接受被取消了的事件。实现方法是在@SubscribeEvent注解或#addListener方法中设置receiveCanceled布尔值为真

3.5 三态与结果

尽管事件处理方法不能有返回值，但是有的事件本身具有不同的TriState或Result枚举类，可以通过某些#set*方法设置事件的结果或“返回值”

Minecraft NeoForge 开发环境配置

发表于 2025-04-27 
本文采用环境：NeoForge 21.5.42-beta，支持Minecraft 1.20.5

0. 准备工作

  • JDK21 (Minecraft 1.20及以上版本使用)
  • IntelliJ IDEA
  • NeoForge MDK

1. 导入项目

  • 从NeoForge官网下载定制MDK模板后，用IDEA打开，将自动导入Gradle项目（如果没有，尝试运行gradlew。此时将下载Gradle并反编译Minecraft，国内网络环境需要科学上网
  • 提前在文件-项目结构中将项目SDK修改为21，文件-设置-构建、执行、部署-Gradle中将Gradle JVM修改为项目SDK

2. 修改gradle.properties

  • 修改mod_id、mod_name等基本信息，注意修改mod_group_name后Mod主类要放在相应目录
  • NeoForge已默认使用Parchment映射，只需手动更新parchment_minecraft_version和parchment_mapping_version即可，版本号可在Parchment文档查询
  • 修改完成后，点击右上角刷新Gradle设置

3. 修改Mod主类

  • 修改Mod主类名称，移动到mod_group_name对应的目录中
  • 个人习惯：将字段MODID更名为MOD_ID
  • 修改MOD_ID，注意只能使用小写英文字母和下划线，作为mod的唯一标识命名空间
    • IDEA重命名快捷键：Shift + F6
  • 删除多余的示例代码，可只留下MOD_ID和空的构造函数

4. 尝试运行

  • 运行Gradle任务Client或runClient，查看是否正常
